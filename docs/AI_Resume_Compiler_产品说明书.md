## 1. 产品是什么

### 1.1 产品定义

**AI Resume Compiler** —— 把求职简历当作可编译的资产：用户维护一份完整母版，针对不同岗位编译出不同的投递版本。

### 1.2 核心心智

- **母版（Master）**：用户所有经历的完整集合，永远完整保留
- **子版（Compiled Version）**：针对具体岗位编译出的精简优化版本
- **编译器（Compiler）**：母版 + JD → 子版

### 1.3 目标用户

按"母版意识"分层：

| 用户类型 | 占比 | 特征 | 进入路径 |
|---------|------|------|---------|
| A 类 | 30% | 有完整简历的成熟职场人 | 直接优化流程 |
| B 类 | 50% | 简历已针对某次投递精简过，没母版意识 | 优化中提示补充母版 |
| C 类 | 20% | 应届生或半成品简历 | 反向引导建母版 |

### 1.4 核心价值主张

针对**每个具体岗位**，从用户的完整经历中**取舍 + 改写**出最优投递版本，同时让用户**清楚知道哪些是 AI 改写、哪些是 AI 推测**。

### 1.5 差异化定位

| 维度 | 行业普遍做法 | 本产品做法 |
|------|------------|-----------|
| 优化方式 | 单次改写 | 母版-子版的持续编译 |
| AI 改写可信度 | 被动免责声明 | 主动信息来源标注 |
| 取舍决策 | 用户自己删 | AI 标记 + 用户确认 |
| 中文场景适配 | 翻译界面 | 适配在线表单投递 |
| 应届生引导 | 通用模板提问 | JD 驱动针对性挖掘 |

## 2. 用户全流程

### 2.1 总览流程图

```
[首次使用]
    │
    ▼
①上传/输入简历 ─────────────────┐
    │                            │
    ▼                            ▼
②AI识别用户类型(A/B/C)     [C类：反向引导建母版]
    │                            │
    ▼                            │
③保存为母版（后台静默）       ─────┘
    │
    ▼
④输入目标JD
    │
    ▼
⑤AI双重分析（取舍 + 改写）
    │
    ▼
⑥过渡页：展示分析结果摘要
    │
    ▼
⑦改写工作台（核心交互）
    │
    ▼
⑧实时反馈（匹配度、页数、关键词）
    │
    ▼
⑨红色"缺失补充"的确认弹窗（如有）
    │
    ▼
⑩完成页：诚实差距 + 应对策略
    │
    ▼
⑪三层导出（PDF / Word / 复制）
    │
    ▼
⑫"我投了这个"标记（可选）
    │
    ▼
⑬子版归档进简历库


[后续使用]
直接从④开始：输入新JD → 基于母版生成新子版
```

### 2.2 关键节点详解

#### 节点 ①：上传/输入

**支持格式**：PDF、Word(.docx)、图片(OCR)、纯文字粘贴

**额外可选**：单独上传头像（中文模板需要）

**用户分流问询**（1 题微调查，5 秒回答）：

```
你正在……
○ 跳槽到下一个岗位
○ 第一份正式工作（应届生）
○ 换行业转型
○ 只是想优化一下简历
```

#### 节点 ②：AI 识别用户类型

后台静默判断依据：
- 字数（母版通常 1.5-2 页内容量）
- 经历数量
- 描述详细程度
- 时间跨度

**用户不需要知道这个判断的存在，但产品流程根据判断结果而不同。**

#### 节点 ⑤：AI 双重分析

并行执行两个任务：

**任务 A：内容相关性评估（取舍）**

对每段经历评估对 JD 的相关性：
- 🟢 强相关：建议保留
- 🟡 弱相关：可保留可隐藏
- 🔴 不相关：建议本次隐藏（不影响母版）

**任务 B：表达优化建议（改写）**

对保留的段落生成改写建议，并标注信息来源：
- 🟢 直接来源：基于原简历
- 🟡 推断信息：基于上下文推测
- 🔴 缺失补充：AI 推测用户可能有但未写

#### 节点 ⑥：过渡页（3 秒停留）

```
✨ 分析完成

针对【字节跳动 - 产品经理】岗位

匹配度    页数      建议改写
65/100   1.4 页    7 处

[开始优化 →]
```

**用途**：让用户在 3 秒内 get 到"产品做了什么"。

#### 节点 ⑦：改写工作台（核心交互）

**布局**：三栏

```
┌──────────────┬──────────────────────┬──────────────┐
│   左栏       │    中栏              │    右栏      │
│ 段落导航     │  当前段落工作区        │  实时反馈    │
│              │                      │              │
│ 段落1 ✓      │  原文 vs AI改写       │ 匹配度: 72   │
│ 段落2 ●      │  改写说明 (为什么)    │ 页数: 1.2    │
│ 段落3 ○      │  信息来源标签          │              │
│ 段落4 ○      │                      │ 关键词命中:  │
│ ...          │  [接受][编辑][跳过]    │ ✓ 数据分析   │
│              │                      │ ○ 跨部门     │
└──────────────┴──────────────────────┴──────────────┘
[进度: ●●●●○○○○ 4/8 · 还需约 3 分钟]
```

**关键交互**：
- 自动定位到第一个待处理段落
- 每段三个动作：接受/编辑/跳过
- 接受后自动跳下一段
- 实时反馈带动画跳变

**首次进入特殊设计**：左右栏暗化，只有中栏亮。第一次点"接受"时，左右栏的动画反馈完成自然 onboarding。

#### 节点 ⑨：红色"缺失补充"的确认弹窗

**触发条件**：用户尝试接受标注为 🔴 的改写建议时

```
⚠️ 需要你确认：
这段经历你原简历里没有，
是 AI 根据 JD 推测你可能有的。

你真的有这段经历吗？
○ 有，确认采纳
○ 没有，本次删除
○ 类似但不完全准确，让我重写
```

**作用**：避免用户不小心采纳了 AI 编造的内容。

#### 节点 ⑩：完成页

```
🎉 优化完成

你刚才做了 5 个改写决策、隐藏了 2 段不相关经历
匹配度从 65 → 84（+19）
同岗位前 25% · 建议投递

[简历PDF实时预览]

⚠️ 仍有 2 项简历无法解决的差距
① JD 要求 5 年管理，你 2 年
  → 面试中强调跨部门协作
② JD 要求 PMP 认证，你没有
  → 加分项，不影响投递

[📥 PDF] [📥 Word] [📋 复制全文] [📋 逐段复制]

[✓ 我投了这个岗位] ← 一键标记，闭环
```

#### 节点 ⑬：子版归档

每个生成的子版自动归档到"我的简历库"，标记：
- 对应岗位（公司 + 职位）
- 生成日期
- 匹配度
- 是否已标记"投了"

用户下次再来，可以基于历史子版继续编辑。

## 3. 功能清单

### 3.1 核心功能（P0，必须做）

#### F1：多格式简历上传与解析
- **输入**：PDF / Word / 图片 / 纯文字
- **输出**：结构化简历数据（姓名、经历、技能等字段）
- **额外**：单独上传头像（可选）
- **工程量**：基础已有，需补强应届生半成品识别

#### F2：母版-子版数据机制
- **母版**：用户完整经历集合，本地存储
- **子版**：每次编译生成的针对性版本，归档到简历库
- **核心约束**：母版永不被覆盖，子版从母版衍生
- **工程量**：约 2 天（原 3 天，借鉴 Resume-Tailor-AI 的 Fact Bank 数据结构节省 1 天）

#### F3：AI 双重分析（取舍 + 改写）
- **取舍**：每段经历对 JD 的相关性评估（🟢🟡🔴 三色）
- **改写**：保留段落的表达优化建议
- **同步执行**，结果合并展示
- **工程量**：约 3 天（原 4 天，借鉴 resumejob/awesome-resume 例句作为 few-shot examples 节省 1 天）

#### F4：三栏改写工作台
- **左栏**：段落导航 + 进度状态
- **中栏**：当前段落原文 + AI 建议 + 控制按钮
- **右栏**：实时匹配度、页数、关键词命中
- **工程量**：约 3.5 天（原 5 天，用 react-diff-viewer-continued 实现 diff 显示节省 1.5 天）

#### F5：信息来源三级标注体系
- 🟢 直接来源 / 🟡 推断信息 / 🔴 缺失补充
- 🔴 类必须用户主动确认才能采纳
- 标注作为 AI 输出的强制字段
- 用 Gemini 2.0+ responseJsonSchema 强制结构化输出
- **工程量**：约 3 天（原 3.5 天，原生 JSON Schema 节省 0.5 天）

#### F6：实时反馈系统
- 匹配度数字动画跳变
- 关键词命中实时更新
- 页数实时预估
- **加权评分系统**借鉴 Resume-Tailor-AI（Hard Skills 2x / Title 1.5x / Context 1x）
- **工程量**：约 2.5 天（原 3 天，借鉴评分算法节省 0.5 天）

#### F7：诚实差距分析
- 区分表达性差距（可改）vs 实质性差距（不可改）
- 实质性差距给出应对策略
- 完成页显著展示
- **借鉴 Resume-Tailor-AI 的关键词分类**（Hard Skills / Title / Business Context / Hard Filters）
- **工程量**：约 2 天

#### F8：智能用户分流
- A/B/C 类自动识别
- B 类：温和提示补充母版
- C 类：进入应届生引导流程
- **工程量**：约 1 天（识别 Prompt）

#### F9：应届生反向引导
- JD 驱动的针对性问题
- 每问题给 4-5 个具体例子
- 口语化回答自动转 STAR 格式
- **完全自做**（这是核心差异化，开源生态无现成方案）
- **工程量**：约 5 天

#### F10：统一 HTML 模板
- 中文带头像版 / 英文无头像版
- 所有用户标准化到此模板
- 编辑过程在 HTML 上完成
- **中文模板基础**：geekcompany/ResumeSample
- **英文模板基础**：OpenResume 或 Reactive Resume 模板
- **工程量**：约 2.5 天（原 5 天，基于现成模板改造节省 2.5 天）

#### F11：三层导出
- PDF：浏览器原生 `window.print()` + react-to-print 优雅封装
- Word：html-docx-js 或 TurboDocx/html-to-docx 库
- 复制：浏览器原生 `navigator.clipboard` API
- **工程量**：约 1 天（原 3 天，三个导出库都现成可用节省 2 天）

#### F12：简历库 + 投递标记
- 母版存储
- 子版归档 + 元信息（岗位、日期、匹配度）
- **投递标记**：用户点"我投了"，子版加 `applied: true`
- 本地存储（localStorage），不接 Firebase
- **借鉴 Resume-Tailor-AI 的 ApplicationsLog 架构**
- **工程量**：约 2 天（原 3 天，借鉴存储架构节省 1 天）

#### F13：渐进式 Onboarding
- 上传后过渡页（3 秒）
- 工作台首屏聚焦中栏
- 第一次"接受"的魔法时刻
- localStorage 标记，只触发一次
- **完全自做**（这是核心差异化，不用 tutorial 类库）
- **工程量**：约 6 天

### 3.2 功能依赖关系

```
F1(上传) ──┬─→ F8(分流) ─┬─→ A类 ──→ F3(分析) ──→ F4(工作台)
           │             │
           │             └─→ C类 ──→ F9(应届生) ──→ F3 ──→ F4
           │
           └─→ F10(模板) 始终被使用

F4(工作台) ─┬─→ F5(信息来源标注) 嵌入展示
            ├─→ F6(实时反馈) 实时更新
            └─→ F7(诚实差距) 完成时展示

F4 完成 ──→ F11(导出) ──→ F12(归档+投递标记)

F13(Onboarding) 包裹 F4 的首次体验
```

## 4. 边界清单（明确不做的事）

### 4.1 被砍掉的功能 + 砍掉理由

| 砍掉的功能 | 砍掉理由 |
|----------|---------|
| 完整投递追踪 / 求职 CRM | 偏离主轴，是 Huntr/Teal Job Tracker 的赛道 |
| 用户使用次数统计/分析 | 不服务功能，只服务运营，作品集不需要 |
| 头像合规检测 / 一键去背景 / 智能裁切 | 不在编译链路上 |
| Firebase 云端同步 / 跨设备 | 作品集不需要，localStorage 够 |
| 完整对话式 AI Agent | Rezi 已做，抄不出彩 |
| AI 协助声明导出（可选项） | demo 看不见，列为 V2 延伸思考 |
| 完整诚信检查页面 | 简化为 🔴 的弹窗确认 |
| 商业化 / 付费墙 / 订阅设计 | 这是作品集 |
| 浏览器插件 / 多端同步 | 功能膨胀 |
| 面试辅导 / 求职社群 / 职业规划 | 偏离主轴 |
| 多 experience level 适配 | 简化为应届生 vs 职场人两种 |
| 大型 examples 库 | Teal/Zety 已做，工程量大 |
| 保留用户原 Word 版式 | 技术上不可行，行业共识 |
| RAG 知识库 | 不是检索问题 |
| Agent 编排 | 不是多步决策问题 |
| 长期留存机制 | 求职是事件驱动，不该追求 |
| 面试提醒/跟进通知 | 投递追踪范畴，砍 |

### 4.2 砍功能的判断标准

任何新功能要满足三个条件才做：

1. **demo 5 分钟里能被看见**
2. **直接服务核心主轴**（母版→针对岗位生成简历）
3. **工程性价比合理**

任一条件不满足，砍。

### 4.3 "投递标记" vs "投递追踪" 的区分

这是容易混淆但必须区分的两件事：

| 维度 | 投递标记（✅ 做） | 投递追踪（❌ 不做） |
|------|---------------|----------------|
| 用户操作 | 点一下"我投了" | 录入公司、岗位、薪资、面试时间... |
| 数据存储 | 在子版上加 `applied: true` | 单独的 application 表 |
| 列表展示 | 简历库子版后显示"✓ 已投" | 独立的"投递管理"页面 |
| 状态追踪 | 无 | 已投/面试/Offer/挂 |
| 提醒通知 | 无 | 跟进提醒 |
| 工程量 | 1 小时 | 5-7 天 |

**只做投递标记，不做投递追踪。**

## 5. 核心交互细节

### 5.1 关键文案规范

| 场景 | ❌ 错误说法 | ✅ 正确说法 |
|------|-----------|-----------|
| 标记不相关经历 | "不相关，建议删除" | "本次投递可隐藏（不影响母版）" |
| AI 改写说明 | "我们改进了你的简历" | "改了什么：xxx；为什么：对应 JD 的 xxx" |
| 实质性差距 | "你缺少 X 能力" | "JD 要求 X，简历无法解决，建议：xxx" |
| 应届生引导 | "请填写你的工作经验" | "你帮老板做过报表吗？社团统计过经费吗？" |
| 上传提示 | （无） | "我们会把简历标准化到模板，确保 ATS 可读" |

### 5.2 关键交互规则

**规则 1：所有删除都是隐藏**
- 任何"删除经历"操作都不真的删除
- 仅在当前子版隐藏，母版保留
- 用户可一键恢复

**规则 2：🔴 类必须主动确认**
- 任何标注为 🔴 的 AI 改写
- 默认不采纳
- 用户必须主动点"确认采纳"才能进入子版

**规则 3：所有改写都附改写理由**
- 不是"AI 替你改好了"
- 而是"改了什么：xxx；为什么：xxx；对应 JD：xxx"

**规则 4：进度始终可见**
- 工作台顶部有进度条
- 显示当前位置 + 预计剩余时间

**规则 5：Onboarding 只触发一次**
- localStorage 标记
- 用户首次和工作台产生交互时标记
- 后续永不重复

### 5.3 关键设计原则

1. **不替用户决策，帮用户看清代价**：AI 给建议、给信息、给推理，用户做决定
2. **诚实优于讨好**：AI 的边界要明确，做不到的事直说
3. **顺应本能，不教育认知**：用产品流程引导用户，不用 tutorial 教育用户
4. **可解释 > 准确**：用户能理解 AI 的推理过程，比 AI 输出准确更重要
5. **保留掌控感**：用户随时能反悔、跳过、自己改

## 6. 技术架构

### 6.1 技术栈

**前端框架（已有）**：

- Motion (动画)

**AI 与文件处理（已有 + 新增）**：

**新增 npm 包（基于开源调研）**：

- `html-docx-js` 或 `@turbodocx/html-to-docx` —— Word 导出
- `react-to-print` —— PDF 打印优雅封装（基于浏览器原生 window.print）

**后端**：
- Cloudflare Pages Functions（API 代理，保护 API Key）

**数据存储**：
- localStorage（母版 + 子版 + onboarding 标记）
- **不接 Firebase**（作品集不需要）

### 6.2 AI 任务清单

| 任务 | 输入 | 输出 | 用途 | 实现方式 |
|------|------|------|------|---------|
| 简历类型识别 | 上传内容 | A/B/C 分类 | 用户分流 | Gemini + JSON Schema |
| 内容相关性评估 | 母版 + JD | 每段 🟢🟡🔴 | 取舍建议 | Gemini + JSON Schema |
| 段落改写 | 原文 + JD | 改写版 + 改写理由 + 信息来源 | 表达优化 + 可信度 | Gemini + JSON Schema（合并任务） |
| 差距分析 | 母版 + JD | 表达性 + 实质性差距 | 诚实告知 | Gemini + JSON Schema |
| 应届生提问 | JD | 5-10 个针对性问题 | 反向引导 | Gemini + JSON Schema |
| STAR 转换 | 用户口语回答 | STAR 格式 bullet | 应届生输出 | Gemini + JSON Schema |

**注**：段落改写 + 信息来源标注合并为一个 AI 任务（不能分开做，标注是改写过程的元数据）。

### 6.3 为什么不用 RAG / Agent

- **不用 RAG**：知识库就是用户自己的母版，不需要外部检索
- **不用 Agent**：简历编译是一次性结构化任务，不是多步决策
- **用什么**：LLM + 状态存储 + JSON 编译器。简单、确定、可调试

### 6.4 数据结构（核心）

**母版（Master）**：

```typescript
{
  userId: string,
  basicInfo: {
    name: string,
    email: string,
    phone: string,
    avatar?: string  // 头像 base64 或 URL
  },
  segments: [
    {
      id: string,
      type: '工作经历' | '项目经历' | '教育背景' | '技能' | ...,
      title: string,
      content: string,
      bullets: string[],
      timeRange: { start: string, end: string },
      tags: string[]  // 用户/AI 打的标签，用于相关性判断
    }
  ]

```

**子版（CompiledVersion）**：

```typescript
{
  id: string,
  masterId: string,        // 关联的母版
  jobInfo: {
    company: string,
    position: string,
    jdText: string
  },
  generatedAt: timestamp,
  applied: boolean,        // 是否已标记"我投了"
  appliedAt: timestamp | null,  // 标记时间
  matchScore: {
    initial: 65,
    final: 84
  },
  segmentDecisions: [
    {
      segmentId: string,           // 关联母版中的 segment
      action: 'keep' | 'hide' | 'rewrite',
      rewrittenContent?: string,   // 改写后的内容
      source: 'green' | 'yellow' | 'red',  // 信息来源标注
      userConfirmed: boolean       // 🔴 类是否经用户确认
    }
  ]
}
```

**Onboarding 状态**：

```typescript
// localStorage 中的标记

  'onboarded_v1': 'true'  // 只标记是否第一次，不记录次数
}
```

## 7. 产品决策依据

### 7.1 关键决策与依据

| 决策 | 依据 |
|------|------|
| 采用母版-子版心智 | Teal 的"衣柜"模型验证（行业领先做法） |
| 不保留用户原版式 | 头部产品共识（Jobscan/Teal/Rezi 都套自家模板） |
| 信息来源三级标注 | 86% 招聘经理认为 AI 让简历夸大太容易，80% 会丢弃 AI 简历 |
| 渐进式 Onboarding | Teal 用 10 步 tutorial 被吐槽 "overwhelmed" |
| JD 驱动的应届生提问 | Teal/Rezi 用通用模板，应届生想不起来；JD 驱动让用户立刻能联想 |
| 逐段独立复制功能 | 中国求职场景大量是 BOSS 直聘等在线表单投递，PDF 投递只是少数 |
| localStorage 而非 Firebase | 作品集不需要跨设备，省工程量 |
| 不用 RAG/Agent | 不是检索/多步决策问题，强行用会引入不确定性 |
| 不做长期留存 | 求职是事件驱动场景，不该追求"日活" |
| 投递标记不做追踪 | 标记是闭环最小成本，追踪是另一个产品 |

### 7.2 与竞品的差异化定位

| 维度 | Jobscan | Teal | Rezi | 本产品 |
|------|---------|------|------|-------|
| 核心定位 | ATS 匹配工具 | 求职管理平台 | ATS 内容优化 | Resume Compiler |
| 母版机制 | 无 | "衣柜"（隐性） | 部分 | 显性的母版-子版 |
| AI 改写可信度 | 被动免责 | 被动免责 | 被动免责 | **主动来源标注** |
| 取舍交互 | 用户自删 | 勾选+AI辅助 | 用户自删 | **AI 标记+用户确认** |
| Onboarding | Tutorial | 10步教程 | 分步引导 | **渐进式自然引导** |
| 应届生支持 | 弱 | 通用模板 | 通用模板 | **JD 驱动挖掘** |
| 中文场景 | 无 | 无 | 无 | **逐段复制等本地化** |

### 7.3 产品哲学

**1. 简历是可编译的资产，不是静态文件**

用户维护一份母版，按需编译出不同版本。简历从"文件"变成"系统"。

**2. AI 是协作者，不是替代者**

AI 给建议、给推理、给选项，用户做决策。AI 的边界用户必须能看到。

**3. 透明度优于功能完整度**

让用户看到 AI 怎么想、为什么这么想、什么地方不确定——比让 AI 看起来无所不能更重要。

**4. 顺应用户本能，不教育用户认知**

产品流程引导用户，让用户在使用中自然形成新认知。不用 tutorial。

**5. 做减法的勇气**

做这一件事到极致，比做十件事每件 60 分更有价值。

## 8. 开源资源借鉴清单

> 这一章是基于 9 轮系统性调研整理的"作战地图"。
> 每个功能模块明确：从哪借、借什么、不借什么。

### 8.1 借鉴原则

**站在巨人肩膀做基础设施，把工程时间花在差异化上。**

凡是"业界标准做法"的工程问题（数据结构、UI 组件、文件转换），借鉴现成方案；
凡是构成产品差异化的部分（信息来源标注、应届生引导、Onboarding 哲学），完全自做。

### 8.2 资源清单（按功能模块）

#### F2 母版-子版数据机制
- **借鉴**：[Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 的 Fact Bank 数据结构
- **如何用**：参考其 `lib/types.ts` 和 `lib/storage.ts`，重写为我们的母版-子版 TypeScript 类型
- **不借鉴**：他用 OpenAI，我们用 Gemini

#### F3 AI 双重分析
- **借鉴 1**：[resumejob/awesome-resume](https://github.com/resumejob/awesome-resume) 的简历例句库
- **如何用**：从中精选 5-10 条优质中文 bullet 作为 few-shot examples 嵌入 Prompt
- **借鉴 2**：[Resume-Matcher](https://github.com/srbhr/Resume-Matcher) 的三层匹配理论
- **如何用**：写入 Prompt 规则——"保留 JD 原词，不要做同义替换"
- **借鉴 3**：[Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 的 minimal edits 原则
- **如何用**：写入 Prompt 约束——"优先保留原文表达，只在必要时改写"

#### F4 三栏改写工作台
- **借鉴**：[react-diff-viewer-continued](https://www.npmjs.com/package/react-diff-viewer-continued)（npm 包）
- **如何用**：`npm install react-diff-viewer-continued`，在中栏显示"原文 vs 改写版"的 diff 视图
- **借鉴**：[Reactive Resume v5.x](https://github.com/AmruthPillai/Reactive-Resume) 的 AI Proposal Queue 设计模式
- **如何用**：学他的产品交互模式（accept/reject 队列），不抄代码

#### F5 信息来源标注
- **完全自做** —— 这是核心差异化
- **借鉴**：[Resume-Tailor-Agents](https://github.com/Soroush-aali-bagi/resume-tailor-agents) 的 No Hallucinations Policy
- **如何用**：作为我们 Prompt 中"红色缺失补充"必须标注的约束基础

#### F6 实时反馈系统
- **借鉴**：[Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 的加权 ATS 评分系统
- **如何用**：直接照搬权重——Hard Skills 2x / Title 1.5x / Business Context 1x
- **借鉴**：[Resume-Matcher](https://github.com/srbhr/Resume-Matcher) 的关键词归一化
- **如何用**：写入 Prompt 规则——"React.js / ReactJS / React 视为同一关键词"

#### F7 诚实差距分析
- **借鉴**：[Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 的关键词分类
- **如何用**：借鉴其 Hard Filters 概念，区分"硬差距"（一票否决）和"软差距"（优化项）

#### F8/F9 用户分流 + 应届生引导
- **完全自做** —— 这是核心差异化
- 开源生态没有 JD 驱动反向引导的应届生方案

#### F10 统一 HTML 模板
- **借鉴 1（中文）**：[geekcompany/ResumeSample](https://github.com/geekcompany/ResumeSample)
- **如何用**：选一个简洁中文模板，HTML/CSS 改造为可数据驱动的 React 组件，加头像位
- **借鉴 2（英文）**：[OpenResume](https://github.com/xitanggg/open-resume) 或 [Reactive Resume](https://github.com/AmruthPillai/Reactive-Resume) 的模板
- **如何用**：选一个无头像的 ATS-friendly 模板做基础

#### F11 三层导出
- **借鉴 1**：浏览器原生 `window.print()` + [react-to-print](https://www.npmjs.com/package/react-to-print)
- **如何用**：PDF 导出，1 天搞定，文本可选、ATS 可读
- **借鉴 2**：[html-docx-js](https://github.com/evidenceprime/html-docx-js) 或 [TurboDocx/html-to-docx](https://github.com/TurboDocx/html-to-docx)
- **如何用**：`npm install`，直接调用 `asBlob()` 把 HTML 转 DOCX
- **借鉴 3**：浏览器原生 `navigator.clipboard.writeText()`
- **如何用**：实现"逐段复制"功能，零依赖

#### F12 简历库 + 投递标记
- **借鉴**：[Resume-Tailor-AI](https://github.com/JaimeYeung/Resume-Tailor-AI) 的 ApplicationsLog
- **如何用**：参考其存储架构，但简化字段（只要 applied / appliedAt 两个字段）

#### F13 渐进式 Onboarding
- **完全自做** —— 这是核心差异化
- **明确不借鉴**：react-user-onboarding、Shepherd.js 等 tutorial 类库（哲学相反）

### 8.3 关键技术能力借鉴

#### Gemini 2.0+ 原生结构化输出
- **能力**：用 `responseJsonSchema` 参数强制 LLM 输出符合 JSON Schema
- **用途**：所有 AI 任务都用此方式，避免脆弱的字符串解析
- **官方文档**：https://ai.google.dev/gemini-api/docs/structured-output
- **节省工时**：1 天（不用写错误处理、retry 机制）

### 8.4 借鉴 vs 自做的判断矩阵

| 类型 | 借鉴 | 自做 |
|------|------|------|

| UI 组件库 | ✅ react-diff-viewer | |
| 文件转换 | ✅ html-docx-js | |

| Prompt examples | ✅ resumejob | |
| 算法/约束理论 | ✅ Resume-Matcher + Resume-Tailor-AI | |
| **核心交互流** | | ✅ 自做 |
| **信息来源标注** | | ✅ 自做 |
| **应届生 JD 驱动引导** | | ✅ 自做 |
| **渐进式 Onboarding** | | ✅ 自做 |
| **中文场景适配（逐段复制等）** | | ✅ 自做 |

**借鉴节省**：约 13.5 天

### 8.5 与所有竞品的关键差异（再次明确）

调研结论强化了一个产品判断：

**所有现有简历工具都把"AI 改写的可信度"作为被动免责声明**（Teal、Jobscan、Rezi 全都是）。

我们做了相反的事：**主动信息来源标注**。

这是开源调研后**唯一确认的市场空白**，也是这个项目最值得花时间的地方。
